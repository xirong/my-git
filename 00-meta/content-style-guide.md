# 内容风格规范

这个文档约束后续所有新文章，避免仓库内容停留在链接合集。

## 每篇文章要回答的问题

1. 这篇解决什么实际问题
2. 什么情况下应该用
3. 什么情况下应该避免
4. 安全做法是什么
5. 高风险操作是什么
6. 底层心智模型是什么

## 推荐结构

```text
# 标题

## 问题

## 适合场景

## 推荐做法

## 命令

## 示例

## 常见错误

## 延伸阅读
```

## 写作原则

- 用工程判断组织内容，不堆砌资料链接
- 先讲场景，再给命令
- 涉及 `reset --hard`、`push --force`、历史重写等操作时，必须写清风险
- 涉及团队协作时，必须区分“本地未提交”“已提交未 push”“已 push”“已被别人基于它开发”
- 涉及 GitHub 功能时，优先引用官方文档
- 不把某一种工作流包装成所有团队都适用的标准答案
- 旧资料可以保留，但要标注时效性
